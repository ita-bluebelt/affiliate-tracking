Step-by-Step Description

8 processing stages for every event

1

Event Ingestion

BuzzNet client calls POST /events or /events/batch. A UUID-v4 event_id is generated for each event.

2

Authentication

Request authenticated via API key or OAuth2. Unauthorized requests are rejected with 401.

3

Schema Validation

Event schema validated against JSON/Avro definition. Invalid events rejected with 400.

4

Metadata Enrichment

Metadata attached: ip, session_id, source, user_agent, schema_version.

5

Fraud Signal Detection

4 checks run in parallel. Events are always recorded — checks only set flags.

is_duplicate anomaly_detected bot_suspected ip_flagged
6

Build Fraud Signals

All detection results aggregated into a single fraud_signals object attached to the event.

7

Sign & Publish

Event signed with HMAC-SHA256 for tamper-proof integrity, then published to Kafka partitioned by user_id.

8

Store & Monitor

Kafka consumer writes to ClickHouse (append-only). If fraud signals present → alert triggered. Event stored regardless.

Risk Analysis & Mitigation

Security, integrity and operational risk controls

🔐 Gateway Security
  • Mandatory authentication (API key / OAuth2)
  • Rate limiting per user/IP
  • IP & User-Agent filtering
  • Server-side tracking only — no client-only events
🔍 Processing Logic
  • Idempotency keys to prevent duplicate recording
  • Smart deduplication at ingestion layer
  • Anomaly detection for rapid consecutive events
  • Bot pattern recognition
🗄️ Data Integrity
  • Append-only ClickHouse — no updates or deletes
  • HMAC-SHA256 signature on every event
  • Full audit traceability
  • Kafka 7-day retention for replay
📋 Compliance & IPO Readiness
  • Log integrity — tamper-proof records
  • Traceability — full audit trail per user
  • Evidence-grade dispute resolution
  • Audit report export (PDF / CSV)
🚀 Future Enhancements
  • Device attestation
  • ML-based bot detection
  • WAF integration
  • AI analysis & optimization layer
⚖️ Core Principles
  • Never reject events at tracking layer (except auth/schema)
  • Separation: Tracking ≠ Scoring ≠ Rewards
  • No points, no evaluation — record facts only
  • IDs standardized across all systems
