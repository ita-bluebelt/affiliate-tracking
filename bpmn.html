Step-by-Step Description

8 processing stages for every event

1

Event Ingestion

BuzzNet client calls POST /events or /events/batch. A UUID-v4 event_id is generated for each event.

2

Authentication

Request authenticated via API key or OAuth2. Unauthorized requests are rejected with 401.

3

Schema Validation

Event schema validated against JSON/Avro definition. Invalid events rejected with 400.

4

Metadata Enrichment

BuzzNet API Service attaches metadata before sending S2S to Tracking API: ip, session_id, source, user_agent, schema_version.

5

Fraud Signal Detection

4 checks run in parallel. Events are always recorded — checks only set flags.

is_duplicate anomaly_detected bot_suspected ip_flagged
6

Build Fraud Signals

All detection results aggregated into a single fraud_signals object attached to the event.

7

Sign & Publish

Event signed with HMAC-SHA256 for tamper-proof integrity, then published to Kafka partitioned by user_id.

8

Store & Monitor

Kafka consumer writes to ClickHouse (append-only). If fraud signals present → alert triggered. Event stored regardless.

Fraud Detection

3 parallel checks — detect and flag only, never reject or drop events

⚑
Core Principle
Fraud Detection never rejects events (except auth/schema failures). It only attaches flags to the event before writing to ClickHouse. The backend layer decides how to handle flagged events.
is_duplicate anomaly_detected anomaly_reason severity bot_suspected

1 · Duplicate Event

checkDuplicate(event)

🔁
What is this fraud?

Sending the same event multiple times to inflate metrics. Example: submitting task_completed 10 times for the same task.

How to detect
Idempotency Key =
  hash(user_id + event_type
      + content_id + timestamp_rounded)
Implementation
  • → Redis SET key EX 86400 NX (expire 24h)
  • → Round timestamp to the minute to catch duplicates within short windows
Output flags
"is_duplicate": true,
"duplicate_of": "original_event_uuid"

2 · Anomaly Behavior

checkAnomalyBehavior(userId, eventType)

⚡
What is this fraud?

Actions performed at an abnormal frequency. Example: sharing 100 posts within 60 seconds.

How to detect — Sliding window
Count events of same type per user in 60s
If count exceeds threshold → anomaly
Suggested thresholds
Event Type Max / 60s Severity
task_completed3low
post_created5low
share_executed10medium
comment_created20medium
follow_executed30high
Implementation
  • → Redis Sorted Set: ZADD user:{id}:{type} ts eventId
  • → Use ZCOUNT within the time window to count events
Output flags
"anomaly_detected": true,
"anomaly_reason": "rapid_consecutive_events",
"severity": "high"

3 · Bot Pattern

checkBotPattern(event)

🤖
What is this fraud?

Automated scripts performing actions that look human but exhibit characteristic patterns.

Combined signals
Signal How to check
Headless browser User-Agent contains HeadlessChrome, Selenium
Too regular stddev(timestamps) < threshold
Missing metadata No session_id or referrer present
Invalid sequence task_completed without a preceding task_started
Multi-account / IP Multiple distinct user_ids from the same IP within 1h
Implementation
  • → Score-based: each signal adds to a cumulative score
  • → Total score > threshold → bot_suspected = true
Output flags
"bot_suspected": true,
"bot_reason": "headless_browser" // headless_browser | too_regular |
                      // missing_metadata | invalid_sequence |
                      // multi_account_ip

buildFraudSignals(checks)

Merge results from all 3 checks into a single object attached to the event

{
  "fraud_signals": {
    "is_duplicate":    false,
    "duplicate_of":    null,
    "anomaly_detected": true,
    "anomaly_reason":  "rapid_consecutive_events",
    "severity":        "high",
    "bot_suspected":   false
  }
}
This object is stored in ClickHouse alongside the event. The backend / Guild service reads fraud_signals to decide: exclude from reward calculation, suspend the user, or trigger an admin alert.

Risk Analysis & Mitigation

Security, integrity and operational risk controls

🔐 Gateway Security
  • Mandatory authentication (API key / OAuth2)
  • Rate limiting per user/IP
  • IP & User-Agent filtering
  • Server-side tracking only — no client-only events
🔍 Processing Logic
  • Idempotency keys to prevent duplicate recording
  • Smart deduplication at ingestion layer
  • Anomaly detection for rapid consecutive events
  • Bot pattern recognition
🗄️ Data Integrity
  • Append-only ClickHouse — no updates or deletes
  • HMAC-SHA256 signature on every event
  • Full audit traceability
  • Kafka 7-day retention for replay
📋 Compliance & IPO Readiness
  • Log integrity — tamper-proof records
  • Traceability — full audit trail per user
  • Evidence-grade dispute resolution
  • Audit report export (PDF / CSV)
🚀 Future Enhancements
  • Device attestation
  • ML-based bot detection
  • WAF integration
  • AI analysis & optimization layer
⚖️ Core Principles
  • Never reject events at tracking layer (except auth/schema)
  • Separation: Tracking ≠ Scoring ≠ Rewards
  • No points, no evaluation — record facts only
  • IDs standardized across all systems
